# Журнал технических решений

Файл ведётся по шагам плана: что сделано и почему выбрано именно такое решение.

## Шаг 1. Скелет проекта, Docker, Makefile, .env.example

- Инфраструктура (Makefile, ci.yml, .golangci.yml, multi-stage Dockerfile, compose с healthcheck Postgres, .gitignore, .gitattributes, LICENSE) перенесена из `go-price-tracker` и адаптирована под домен: бинарник `parser` → `bot`, креды `tracker` → `consultant`. Проверенная оснастка — не переписываем ради переписывания.
- `go.mod`: `go 1.25` (а не локальный 1.26.4), чтобы совпадать с `golang:1.25-alpine` в Dockerfile и `go-version: "1.25"` в CI. Go 1.26 собирает модуль, требующий 1.25, без проблем.
- Сертификат Минцифры **не хранится в git** (`certs/*.cer` в .gitignore): это бинарный артефакт, качается локально через `make cert` с единственного разрешённого URL `https://gu-st.ru/.../russian_trusted_root_ca.cer`. В репозитории — только `certs/.gitkeep`, чтобы каталог существовал для `COPY certs/` в Dockerfile. Проверено: PEM, subject/issuer = Russian Trusted Root CA (self-signed корень).
- Dockerfile копирует в образ `knowledge/` и `certs/`: и база знаний, и сертификат читаются с диска на старте. Шаг `make cert` добавлен в CI перед `docker build`, иначе `COPY certs/` упадёт.
- `.env.example` — только плейсхолдеры. Переменные разбиты по блокам; критичные для конкретного `LLM_PROVIDER` секции помечены. `GIGACHAT_INSECURE_SKIP_VERIFY` документирован как крайний фолбэк с явным «никогда в проде».
- В compose у `app` нет volume для вывода (в отличие от price-tracker): консультанту нечего писать на хост, весь вывод — в Telegram и в БД.

## Шаг 2. Конфиг из ENV и slog

- `config.Load()` копит проблемы в срез и возвращает одной ошибкой — за один запуск видно весь список недостающих переменных, а не по одной.
- **Провайдер-специфичная валидация.** Ключи GigaChat обязательны только при `LLM_PROVIDER=gigachat`, ключи Yandex — только при `yandexgpt`. Требовать всё сразу заставляло бы держать в `.env` мусорные значения для неиспользуемого провайдера. Реализовано `switch` по провайдеру с отдельными `loadGigaChat` / `loadYandex`, возвращающими свои проблемы.
- Fail fast на `BOT_TOKEN`, `DATABASE_URL`, корректном `LLM_PROVIDER` и ключах выбранного провайдера. `ADMIN_TG_ID` опционален (пусто → `/reload` недоступна никому), у `SCOPE/MODEL/CERT_PATH/KNOWLEDGE_PATH/APP_ENV` есть дефолты.
- `GigaChatConfig` и `YandexConfig` — вложенные структуры, а не плоские поля: клиент получает ровно свой набор параметров, без чужих.
- `NewLogger(appEnv)` принимает окружение параметром (не читает ENV сам) — вызывается из main после Load и тестируем.

## Шаг 3. LLM-клиент: интерфейс, GigaChat, YandexGPT

- **Интерфейс `LLMClient` намеренно узкий:** `Ask(ctx, systemPrompt, userMessage) (string, error)`. Всё, что нужно боту, — задать вопрос и получить текст; подмена провайдера через ENV сводится к одной строке в фабрике `llm.New`.
- **Свой клиент на net/http, а не SDK.** Два HTTP-вызова (OAuth + chat) не окупают внешнюю зависимость, а свой код проще аудировать в чувствительных местах — TLS-пул и кэш токена. SDK упомянуты в README как опция.
- **CertPool с сертификатом Минцифры.** Цепочка TLS GigaChat подписана корнем НУЦ Минцифры, которого нет в системном хранилище. Клиент читает PEM с диска, добавляет в копию системного пула (`SystemCertPool` + `AppendCertsFromPEM`), ставит `MinVersion: TLS1.2`. `GIGACHAT_INSECURE_SKIP_VERIFY=true` — крайний фолбэк с `log.Warn` и `//nolint:gosec`.
- **Кэш OAuth-токена.** Токен живёт 30 минут; кэшируем на `TTL` минус окно безопасности 60с, обновляем лениво под мьютексом (к боту пишут параллельно). Срок жизни считаем от момента получения (`now().Add(TTL)`), а не парсим `expires_at`: проще, детерминированно тестируется, а дрейф закрывает путь повтора на 401. `now` — поле-функция, тест управляет временем без sleep.
- **401 → ровно один повтор.** `chat` возвращает HTTP-статус отдельным значением; на 401 `Ask` принудительно переполучает токен и повторяет один раз. Второй 401 отдаётся как ошибка — без зацикливания (есть тест `TestGigaChat401NoInfiniteRetry`).
- **Таймаут.** У `http.Client` `Timeout: 0` — дедлайн держит только `context` вызывающего (45с в проде). Короткий глобальный Timeout перебил бы контекст и рвал штатную 10–20с генерацию. Тесты `TestGigaChatTimeout`/`TestYandexTimeout` проверяют, что при дедлайне контекста возвращается `context.DeadlineExceeded`.
- **YandexGPT — вторая реализация за тем же интерфейсом.** Проще: `Api-Key`, публичные CA. Ключевая ловушка вынесена в тип и комментарий: у Yandex поле сообщения — `text`, а не `content`; счётчики токенов приходят строками. `modelUri` собирается как `gpt://<folder>/<model>/latest`.
- Тела ответов читаются через `io.LimitReader(1МиБ)`; ошибки чужих ответов обрезаются `snippet` — незачем тащить в лог мегабайт HTML.
- **TODO:** сверить формат ответа OAuth (`expires_at`, единицы) и chat/усечения ошибок с офиц. документацией developers.sber.ru/docs/ru/gigachat и yandex.cloud/ru/docs/foundation-models — реальные ключи не тестировались (по условию).

## Шаг 4. knowledge: загрузка, чанкинг, отбор, лимит

- **RAG-lite без векторов.** Для FAQ/прайса одной компании эмбеддинги и векторная БД — избыточная инфраструктура. Чанкинг по markdown-заголовкам + keyword-отбор дают приемлемое качество при нулевой оснастке. Осознанный размен, зафиксирован в README.
- **Чанкинг.** `parseSections` режет файл на секции от заголовка (`#…`) до следующего. Текст до первого заголовка и файл вовсе без заголовков становятся одной безымянной секцией — терять его нельзя. Вложенные заголовки → каждый своя секция. Пустые (только пробелы) секции отбрасываются.
- **Отбор с префиксным стеммингом.** Точное совпадение подстроки промахивалось на русской морфологии («оплаты» ≠ «оплата», «картой» ≠ «карты»). Ввёл дешёвый стем — префикс в `stemLen=4` рун: сравниваем стемы слов вопроса и секции. Полноценная лемматизация не нужна для демо, а 4-символьный префикс ловит корень основных падежей. Счёт секции — число совпавших по стему слов.
- **Фолбэк.** Нет ни одного совпадения → берём первые секции файла под лимит, чтобы модель получила общий контекст о компании, а не пустоту (тест `TestSelectFallback`).
- **Лимит размера промпта.** `MaxPromptChars=12000` (~4000 токенов при оценке символы/3). `fitToLimit` набирает секции в порядке релевантности, пока укладываются; `assemble` собирает их обратно в порядке документа (читабельнее) и страхует финальным обрезанием по рунам. Если самая релевантная секция одна больше лимита — берётся и обрезается, промпт не пустой (тест `TestSelectRespectsLimit`).
- Стоп-слова (частые местоимения/предлоги) отсеиваются, иначе «как»/«для» матчили бы каждую секцию и обнуляли ранжирование.
- `Base` потокобезопасен (`sync.RWMutex`): `/reload` подменяет секции, пока параллельные запросы их читают.

## Шаг 5–8. Telegram, логирование, /reload, обработка ошибок

- **Ядро `Consultant` отделено от транспорта.** `Consultant.Answer(ctx, question)` делает всё содержательное: отбор секций, сборка системного промпта, запрос к LLM под таймаутом. Зависит от интерфейсов `LLM`/`Knowledge`, поэтому тестируется заглушками без сети и без Telegram. `Bot` — тонкая обёртка: приём апдейтов, «печатает…», отправка, запись в журнал.
- **Системный промпт** инструктирует отвечать ТОЛЬКО по базе знаний, честно признавать незнание и не выдумывать цены/сроки. Пустая база помечается явно — чтобы не выглядело, будто контекст потерялся.
- **Таймаут 45с** навешивается в `Answer` через `context.WithTimeout` поверх процессного ctx. `TestAnswerAppliesTimeout` проверяет, что отменённый родитель прерывает запрос.
- **`ChatAction "печатает…"`** перед запросом: генерация идёт 10–20с, без индикатора кажется, что бот завис.
- **Логирование диалогов best-effort.** Ошибка `SaveDialog` пишется в лог, но не отменяет уже отправленный пользователю ответ: журнал вторичен по отношению к ответу. Проверено интеграционным тестом на реальном Postgres (`TestSaveDialogIntegration`, под `TEST_DATABASE_URL`).
- **`tokens_used` — оценка (символы/3).** Интерфейс `LLMClient` намеренно узкий и точный `usage` не возвращает; реальный `total_tokens` пишется в debug-лог клиента. Для аналитики журнала грубой оценки достаточно, а колонка `tokens_used` — nullable.
- **`/reload` только для админа.** Сверяем `From.ID` с `ADMIN_TG_ID`; при пустом админе или чужом id — вежливый отказ. Неизвестные команды (`/foo`) не уходят в LLM, чтобы не жечь токены.
- **Ошибки пользователю понятны и без внутренностей.** `userFacingError` отделяет таймаут («попробуйте ещё раз») от прочей недоступности. 401 GigaChat обрабатывается ниже — внутри клиента, один повтор с новым токеном.

## Шаг 10. Сквозной graceful shutdown

- Один `signal.NotifyContext(SIGINT, SIGTERM)` создаётся в `main` и уходит во все блокирующие вызовы: миграции, пул БД, `bot.Start`, каждый запрос к LLM (как родитель 45с-дедлайна). По сигналу останова отменяется всё дерево разом; `bot.Start` возвращает управление, `store.Close()` отрабатывает через defer.
- `Bot.Run` возвращает `ctx.Err()` после остановки polling — процесс завершается штатно, а не по таймауту убийства контейнера.

## Ревью и правки (пост-аудит)

Строгое ревью выявило 9 проблем; исправлены все. Ниже — что изменилось и почему;
эти решения заменяют часть более ранних (помечено «отменяет»).

- **№1 Graceful shutdown завершался кодом 1.** `Bot.Run` возвращал `ctx.Err()` → на каждый SIGINT/`docker compose down` процесс печатал `ошибка: context canceled` и выходил с 1, а `restart: on-failure` считал это сбоем. Теперь `Run` возвращает `nil`, если ctx отменён штатно (`errors.Is(err, context.Canceled)`).
- **№2 `tokens_used` — теперь фактический usage провайдера.** *Отменяет решение Шага 3 про узкий `(string, error)`.* Интерфейс `LLMClient.Ask` возвращает `llm.Answer{Text, TokensUsed}`. Оба клиента прокидывают `usage.total_tokens` (у Yandex — из строки). Оценка символы/3 осталась только фолбэком, когда провайдер usage не вернул, и теперь считает **весь** промпт (системный + вопрос + ответ), а не только вопрос+ответ — иначе расход занижался кратно (важно для 1M-токенного freemium GigaChat).
- **№3 Срок жизни токена — по серверному `expires_at`.** *Отменяет решение Шага 3 «не парсим expires_at».* `fetchToken` читает `expires_at` (Unix-мс), срок = `min(now+30м, серверный expires_at)`. Кэш больше не переживёт реальный токен при коротком сроке/дрейфе часов; повтор на 401 остаётся страховкой.
- **№4 Лок больше не держится на время сетевого запроса.** Быстрый путь читает кэш под коротким мьютексом и отпускает его; само переполучение идёт через `singleflight.Group` — N параллельных промахов (или 401) схлопываются в один поход в OAuth. Раньше один медленный OAuth блокировал все параллельные `Ask`, включая тех, у кого валидный токен. Тест `TestGigaChatConcurrentTokenFetch` (20 горутин → 1 запрос, под `-race`).
- **№5 Добавлен `.dockerignore`.** `COPY . .` тянул в build-контекст и промежуточный слой локальный `.env` с реальными ключами, `.git`, тесты. Теперь секреты и мусор исключены. Docker build перепроверен — зелёный.
- **№6 Лимит промпта — единый источник.** `assemble` принимает `maxChars` параметром вместо хардкода `MaxPromptChars`; теперь и набор, и финальная обрезка используют одно значение (`b.maxChars`).
- **№7 Заголовки внутри code-fence не режут секцию.** `parseSections` отслеживает fenced-блоки (```` ``` ````/`~~~`); строка `# comment` в примере кода больше не создаёт фиктивную секцию. Тест добавлен.
- **№8 Секреты и DSN триммятся.** `BOT_TOKEN`, `DATABASE_URL`, `GIGACHAT_AUTH_KEY`, `YANDEX_API_KEY`, `YANDEX_FOLDER_ID` очищаются от случайных пробелов/переносов при копипасте в `.env` — иначе битый `Authorization: Basic …\n` давал невоспроизводимые 401.
- **№9 Защита границ системного промпта.** `sanitizeKnowledge` вырезает из контента базы строки, совпадающие с разделителями `=== БАЗА ЗНАНИЙ ===` / `=== КОНЕЦ … ===`, чтобы строка из FAQ не подделала границу блока и не «дописала» инструкции модели. Тест `TestSanitizeKnowledgeStripsMarkers`.
- **Проверка:** `go build`, `go vet`, `go test -race ./...`, `make lint` (0 issues), `docker build`, интеграционный тест записи в Postgres — всё зелёное после правок. `x/sync` стал прямой зависимостью (singleflight).
